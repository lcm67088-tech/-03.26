## 1. 개요

네이버 플레이스 SEO에서 **순위키(메인키)의 순위를 올리기 위해 실제 작업에 투입할 키워드(작업키)를 자동으로 발굴**하는 기능.

조합 생성 → LLM 의미 검증 → 파서 순위 조회 → 작업키 확정의 흐름으로 동작한다.

---

## 2. 핵심 개념 정의

| 용어 | 정의 |
|---|---|
| **대표키** | 네이버 플레이스에 등록된 공식 키워드 (`keyword_list`). 조작 불가, 재료 역할 |
| **순위키 (메인키)** | 순위를 올리고 싶은 목표 키워드. 최대 1~3개. 보통 `지역명+업종` or `지역명+메뉴` |
| **작업키** | 순위키 순위 상승을 위해 실제 SEO 작업에 투입하는 롱테일 파생 키워드 |

### 작업키 선별 조건
| 조건 | 기준 | 판정 |
|---|---|---|
| 순위 | 1~5위 | ✅ 작업가능 |
| 순위 | 6~300위 | ⚠️ 모니터링 |
| 순위 | 300위↑ or 미발견 | ❌ 제외 |
| CPC | 없음 선호 | 있으면 경쟁 심함 신호 |
| 플레이스 최상단 | 확인 필요 | 파서에서 함께 반환 |

## 3. 재료 수집

| 재료 | 출처 | 처리 방식 |
|---|---|---|
| **지역키** | 순위키에서 직접 파싱 | 뎁스 확장 양방향 (주로 좁힘) |
| **메뉴/서비스키** | 소개글 + 메뉴 탭 자동 추출 | 토큰화 + 교차검증 + 중복제거 |
| **업종키** | 대표키 + 카테고리 자동 추출 | 사용자 최종 선택 |
| **수식어** | LLM이 소개글/메뉴/대표키 기반으로 동적 생성 | 업종별 고정 없음, 의미 판단 포함 |
| **랜드마크키** | 사용자 직접 입력 | 자동 추출 없음 |

### 메뉴키 처리 방식
- 메뉴명 토큰화: `치즈카츠 정식` → `치즈카츠`, `정식`, `치즈카츠 정식`
- 교차검증: 같은 업체에 `치즈카츠 정식` + `왕돈카츠` → `치즈카츠`, `돈카츠` 계열 클러스터링
- 중복 토큰 제거
- 메뉴 없는 업종(미용실, 피부과 등)은 시술명/서비스명을 메뉴키로 동일하게 처리

## 4. 조합 공식

### 기본 공식
```
[지역/뎁스] + [랜드마크+근처|인근|주변|부근]? + [메뉴/서비스키]? + [업종키] + [수식어]?
```
- **업종키는 항상 맨 끝**
- **수식어는 업종키 뒤에만 붙음**
- **지역은 항상 맨 앞**

### 조합 우선순위 (퍼널 순서)
```
1순위: 지역 + 메뉴 + 업종
       예: 강남 돈카츠 맛집

2순위: 지역 + 뎁스 + 메뉴 + 업종
       예: 강남구 역삼동 돈카츠 맛집

3순위: 랜드마크 조합
       예: 강남 역삼문화공원 근처 맛집
           강남 역삼문화공원 근처 돈카츠 맛집

4순위: 업종키 + 수식어 조합
       예: 강남 돈카츠 맛집 추천
           강남 돈카츠 맛집 가격
           강남 돈카츠 맛집 위치
```

### 수식어 설계 방식
- **하드코딩 없음**
- LLM이 소개글 + 메뉴 + 대표키워드를 컨텍스트로 받아 **의미가 통하는 수식어만** 동적으로 생성
- 기본 수식어 풀 (시작점): `추천`, `가격`, `위치`, `장소`, `예약`, `후기`, `근처`
- LLM이 업체 특성에 맞게 필터링 및 추가

### 순서 변형
- 실제 데이터(CSV 12,379개) 분석 결과 순서 변형 패턴 다수 확인
- 기본은 `지역+메뉴+업종` 순서, 옵션으로 순서 변형 포함 가능

### 순위키 여러 개일 때
- 공통 지역 풀 합산 방식
- `강남 맛집` + `역삼 돈카츠` → 지역키: `강남`, `역삼` 합쳐서 조합

## 5. 퍼널 전략 (핵심)

**원칙: 조건 달성 키워드와 유사할수록 달성 확률이 높다**

```
Step 1. 1순위 샘플 20~30개 조회 (약 2~4분)
           ↓
     작업가능 키워드 발견?
      ↙              ↘
     있음              없음
      ↓                 ↓
Step 2.             2순위로 이동
발견된 키워드        → 반복
주변으로 집중 확장
```

### 발견 키워드 기반 확장 예시
```
"강남 역삼동 맛집" 발견 (3위)
→ 즉시 주변 확장:
  강남 역삼동 맛집 추천
  강남 역삼동 맛집 가격
  강남 역삼동 맛집 위치
  강남 역삼동 맛집 장소
  강남 역삼동 맛집 예약
  강남구 역삼동 맛집
  강남 역삼동 돈카츠 맛집
```

### 0개 케이스 처리
- 발생 빈도: 약 1% (백에 하나)
- 대응:
  1. 6~30위 키워드 모니터링 추천
  2. 랜드마크키 추가 제안
  3. 메뉴키/업종키 변경 제안
  4. 재검색 유도

## 6. LLM 연동 설계

### 활용 목적
- 조합 생성 후 **의미없는 조합 필터링**
- 업체 컨텍스트 기반으로 자연스러운 키워드만 선별
- 수식어 동적 생성

### LLM 판단 흐름
```
입력 컨텍스트:
- 업체명, 카테고리, 소개글, 대표키워드, 메뉴 목록

후보 키워드 100개 배치 전달

LLM 반환:
✅ 유효 (소개글/대표키와 연관성 있음)
⚠️ 불확실 (연관성 낮음)
❌ 제외 (의미 없음 or 어색한 조합)

→ ✅만 파서 순위 조회
```

### 비용 부담
- **플랫폼 부담** (유저 허들 제거)
- 1회 필터링 비용이 미미 (Gemini Flash 기준 약 0.4원)
- 남용 방지는 플랜별 월 사용 횟수 제한으로 통제

### 플랜별 제한 (안)
| 플랜 | 월 사용 횟수 | 사용 모델 |
|---|---|---|
| Free | 3회 | 기본 모델 |
| Starter | 10회 | 기본 모델 |
| Pro | 30회 | 고급 모델 선택 가능 |
| Enterprise | 무제한 | 최고급 모델 |

### LLM 없는 모드 (기본 모드)
- LLM 필터링 없이 전체 조합 생성 → 파서 조회
- 유저가 직접 결과 판단
- UI에 안내 문구 표시:
  ```
  💡 AI 필터링 미사용 중
     의미없는 조합이 포함될 수 있습니다
  ```

### 모델 비교 (2025년 3월 기준)

## 7. 파서 연동 스펙

### 확정 사항
| 항목 | 값 |
|---|---|
| 조회 1회 소요시간 | 평균 5~10초 |
| 반환 데이터 | 순위 + CPC 개수 + 플레이스 섹션 최상단 여부 |
| 최대 조회 횟수 | 500회 (상한선) |

### 현실적 소요시간
```
500회 풀조회: 약 62분 → 비현실적
퍼널 전략 적용 시:
  샘플 조회: 20~30회 (2~4분)
  발견 후 집중 확장: 50~100회 (6~12분)
  목표: 100~150회 안에 작업가능 키워드 확보
```

### 미확인 사항 (구현 전 필수)
- [ ] 파서 API 엔드포인트 및 요청/응답 구조
- [ ] 키워드 순위 조회 시 `place_id` 파라미터 전달 방식

## 8. UI 플로우

```
키워드 탭
└── [🔍 작업키 찾기] 버튼
        ↓
    모달 오픈
    ┌─────────────────────────────┐
    │ Step 1. 재료 입력            │
    │  - 기준 순위키 선택 (체크박스) │
    │  - 지역키 (자동추출+수정가능) │
    │  - 메뉴/서비스키 (자동추출)   │
    │  - 업종키 (자동추출+선택)     │
    │  - 랜드마크키 (직접입력)      │
    │  - 조합 옵션 체크박스         │
    │  [조합 미리보기] [조회 시작]  │
    └─────────────────────────────┘
        ↓
    Step 2. LLM 필터링 (API 있을 때)
    → 의미없는 조합 자동 제거
        ↓
    Step 3. 파서 순위 조회
    → 실시간 진행바
    → 발견 키워드 스트리밍 표시
    → [중지] 버튼
        ↓
    Step 4. 결과 표시
    → 작업가능(1~5위) / 모니터링(6~300위) / 제외
    → 체크박스로 선택
    → [작업키로 추가] 버튼
        ↓
    0개일 때 → 대안 제시 화면
```
